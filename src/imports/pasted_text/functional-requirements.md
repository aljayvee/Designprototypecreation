Functional Requirements Login and Authentication Module: Requirement ID: Functional Specification REQ001,"The system shall display two distinct secure login interfaces: a primary interface for enterprise roles (Owner, Dispatcher, Rider) and a secondary interface exclusively for Customers." REQ002,"The system shall provide an account creation/registration feature strictly on the Customer login interface, and accept standard username and password inputs across both interfaces." REQ003, The system shall be able to validate user credentials against the encrypted relational database. REQ004,"The system shall display an explicit error message for invalid, expired, or suspended login credentials." REQ005,"The system shall redirect authenticated users to their distinctly segregated, role-specific portals (Owner Portal, Dispatcher Portal, Rider Portal, or Customer Portal)." Dashboard Management Module: Requirement ID,Functional Specification REQ006,The system shall be able to display the total number of active delivery personnel currently connected to the network. REQ007,"The system shall be able to display the number of delivery personnel currently categorized as ""Offline"" or unavailable." REQ008,The system shall be able to display the real-time queue of pending errand requests awaiting dispatch. REQ009,The system shall be able to display the aggregate volume of active errands currently in progress. REQ010,The system shall be able to display the accumulated daily business revenue (Sugo Share) generated in real time. REQ011,The system shall be able to update all dashboard numerical data and graphical charts in real time without manual browser refresh. REQ012,The system shall be able to display pending high-value transaction approvals requiring dispatcher verification. REQ013,The system shall be able to notify administrative users of unassigned errands that have exceeded the standard waiting threshold. REQ014,The system shall be able to display the logged-in user's official name and designated systemic role. REQ015,The system shall be able to provide a secure logout option that explicitly terminates the user session and clears local authentication tokens. User Management Module: Requirement ID,Functional Specification REQ016,The system shall be able to display comprehensive user records in a structured administrative table. REQ017,"The system shall be able to add new user accounts, requiring the input of full legal names, contact details, and vehicle specifications." REQ018,"The system shall be able to update existing user account information, including mobile numbers and motorcycle registration data." REQ019,"The system shall be able to activate, suspend, and permanently deactivate user accounts to reflect employment status." REQ020,"The system shall be able to securely assign system roles and corresponding data permissions (Owner, Dispatcher, Rider, Customer) during account creation." Record Management Module: Requirement ID,Functional Specification REQ021,"The system shall be able to display customer demographic records in a searchable, structured table interface." REQ022,"The system shall be able to add new customer profiles, capturing primary contact numbers, complete addresses, and specific navigational landmarks." REQ023,The system shall be able to update existing customer profile information and delivery preferences. REQ024,The system shall be able to archive or delete obsolete customer records to maintain database hygiene. REQ025,The system shall be able to search for customer information dynamically utilizing mobile numbers or partial name matches. REQ026,"The searched customer information shall be instantly displayed on the table, allowing for immediate auto-population of new errand forms." Errand Transaction Management Module: Requirement ID,Functional Specification REQ027,The system shall be able to display a standardized digital intake form for logging new errand requests. REQ028,"The system shall require the classification of the errand as Pabili, Padala, or Bills Payment." REQ029,"The system shall be able to record the customer's specified payment modality (Cash on Delivery, GCash, or Bank Transfer)." REQ030,"The system shall be able to transition the errand status through predefined operational milestones (Arrival at Store, Item Purchased, En Route, Delivered)." REQ031,The system shall allow manual intervention by the Dispatcher to adjust errand parameters in the event of customer modifications. REQ032,The system shall require the delivery personnel to utilize the device camera to capture photographic evidence of original store receipts. REQ033,The system shall require the delivery personnel to capture photographic evidence of the procured physical items prior to delivery transit. REQ034,The system shall link all uploaded photographic artifacts permanently to the specific transaction record in the cloud database. REQ035,The system shall notify administrative dispatchers of anomalous errand durations that exceed standard completion times. REQ036,The system shall alert users if an errand request is canceled by the customer after the procurement phase has already commenced. Rate Management Module: Requirement ID,Functional Specification REQ037,The system shall be able to display an administrative interface for configuring base delivery fees and distance brackets. REQ038,The system shall be able to automatically compute the total base delivery fee utilizing the configured distance parameters. REQ039,The system shall strictly store and track the total purchase value of items procured by the rider. REQ040,The system shall automatically append a 25 PHP surcharge to the transaction total for each additional store location recorded. REQ041,"The system shall dynamically compute and apply a 50 PHP flat commission fee for grocery purchases equal to or below 1,000 PHP." REQ042,"The system shall dynamically compute and apply a 10% variable commission fee for grocery purchases strictly exceeding 1,000 PHP." REQ043,"The system shall be able to display the final, itemized digital invoice on the rider's mobile device, outlining all applied fees and surcharges." REQ044,"The system shall be able to automatically flag any non-COD transaction where the total purchase value exceeds 3,000 PHP." REQ045,The system shall halt the dispatch sequence for flagged high-value transactions until explicit dispatcher payment verification is logged. REQ046,The system shall be able to manage adjustments to commission percentage rates via the Business Owner's administrative panel. REQ047,The system shall be able to accurately display the expected amount to be collected for Cash on Delivery transactions. REQ048,The system shall prevent riders from independently altering the algorithmically calculated commission totals. Dispatch Management Module: Requirement ID,Functional Specification REQ049,The system shall be able to display an interactive geographic map rendering the real-time locations of all active delivery personnel. REQ050,The system shall be able to submit dispatch assignments to specific riders utilizing a centralized digital queuing interface. REQ051,The system shall notify the selected delivery personnel immediately upon the generation of a new dispatch assignment. REQ052,The system shall be able to provide the rider with explicit interface options to accept or decline the dispatch assignment. REQ053,The system shall require the input of a standardized reason code if a rider elects to decline a dispatch assignment. REQ054,"The system shall generate a structured Digital Waybill containing customer name, delivery coordinates, and the exact procurement list." REQ055,The system shall track the temporal progression of the dispatch from assignment through to final delivery completion. Report Management Module: Requirement ID,Functional Specification REQ056,The system shall be able to retrieve all completed transaction and fee data corresponding to a defined operational shift. REQ057,"The system shall be able to automatically compute the total ""Rider Share"" aggregated strictly from base delivery fees." REQ058,"The system shall be able to automatically compute the total ""Sugo Share"" aggregated from all commissions and surcharges." REQ059,The system shall be able to compute and reconcile the total business capital deployed for purchases against total cash collected. REQ060,The system shall be able to generate comprehensive daily sales reports detailing total transaction volumes and service category breakdowns. REQ061,The system shall be able to generate analytical reports illustrating weekly revenue trends and peak operational demand hours. REQ062,"The system shall be able to generate individual rider performance metrics, including successful completion rates and average delivery times." REQ063,The system shall be able to display an exhaustive commission log distinguishing between variable 10% fees and flat 50 PHP fees. REQ064,"The system shall be able to generate digital, itemized end-of-day settlement statements for each active rider." REQ065,"The system shall be able to compile an incident report detailing canceled errands, failed deliveries, and customer disputes." REQ066,The system shall be able to compute the precise multiple-store surcharge revenue generated across the entire fleet. REQ067,"The system shall be able to structure all financial reports into standardized, readable table formats within the administrative dashboard." REQ068,The system shall be able to automatically archive daily settlement reports for longitudinal historical analysis. REQ069,The system shall be able to print physical copies of the generated financial and performance reports directly from the browser. REQ070,"The system shall be able to export analytical reports as digital files (e.g., PDF, CSV) for external accounting documentation." Notification Management Module: Requirement ID,Functional Specification REQ071,The system shall automatically cache the complete Digital Waybill data to the rider's local device storage upon dispatch acceptance. REQ072,The system shall permit the rider to view delivery instructions and navigate the application interface without an active cellular data connection. REQ073,The system shall locally queue all temporal status updates and captured photographic evidence generated during network outages. REQ074,The system shall automatically synchronize queued local data with the remote cloud database upon the restoration of internet connectivity. REQ075,The system shall be able to notify the customer via automated SMS or push notification immediately when a rider is dispatched. REQ076,"The system shall be able to notify the customer automatically when the rider logs the ""Item Purchased"" or ""En Route"" status." REQ077,The system shall be able to notify the assigned rider locally when a high-value transaction verification is completed by the dispatcher. Customer-self Service Module: Requirement ID,Functional Specification REQ078,The system shall provide a dedicated Customer Portal accessible only upon successful login through the customer-specific interface. REQ079,"The system shall provide an interface for customers to create custom service orders specifically for Pabili, Padala, and Bills Payment." REQ080,The system shall be able to automatically generate a digital order slip and digital receipt for the customer upon transaction confirmation. REQ081,The system shall provide an interactive interface for customers to trace the real-time geographic location and operational status of their assigned rider. REQ082,The system shall provide an interface to handle and specify payment modalities directly within the customer portal. Non-functional Requirement: Operational Requirement: Requirement ID,Non-Functional Specification REQN001,The administrative dashboard application shall operate natively on Windows 10 (64-bit) or higher operating systems. REQN002,"The administrative dashboard shall maintain full functional compatibility across modern web browsers, specifically Google Chrome, Mozilla Firefox, and Microsoft Edge." REQN003,The mobile client application shall operate reliably on personal devices utilizing the Android 10.0 operating system or higher. REQN004,"The mobile application footprint shall be optimized to execute smoothly on devices equipped with a minimum of 4GB of RAM and mid-tier processors (e.g., MediaTek Dimensity or Snapdragon 695)." Performance Requirements: Requirement ID,Non-Functional Specification REQN005,The system database queries shall retrieve and render historical customer records within 1 second under stable network conditions to prevent intake bottlenecks. REQN006,The administrative dashboard shall completely load all geographical map assets and initial telemetry markers within 2 seconds. REQN007,The backend computational engine shall execute the complex end-of-day settlement algorithms across 150+ transactions within 5 seconds. REQN008,"The mobile application shall utilize the Fused Location Provider API, intelligently shifting between satellite, Wi-Fi scanning, and cellular triangulation to optimize power draw." REQN009,The mobile application shall dynamically throttle background location polling frequency when device accelerometers detect an extended stationary state. Security Requirements: Requirement ID,Non-Functional Specification REQN010,"The system architecture shall restrict backend access exclusively to authorized, authenticated administrative personnel via strict Role-Based Access Control (RBAC)." REQN011,"The system shall ensure that only users with the ""Business Owner"" role designation have the technical capability to modify base rates, commission percentages, and multiple-store surcharges." REQN012,"The system shall explicitly restrict the ""Dispatcher"" role from executing final approvals on high-value transaction settlements without corresponding digital payment verification logs." REQN013,"All network communications between the mobile client, web dashboard, and the Firebase/MySQL infrastructure shall be encrypted utilizing Transport Layer Security (TLS 1.2 or higher)." REQN014,"The relational database shall secure all stored user passwords at rest utilizing robust cryptographic hashing algorithms (e.g., bcrypt)." REQN015,"The mobile application shall implement a mandatory digital consent mechanism requiring delivery personnel to obtain customer permission prior to capturing geotagged photographic evidence of properties, adhering to the Philippines Data Privacy Act of 2012." REQN016,The system shall automatically obfuscate exact delivery coordinates and sensitive customer contact information from the rider's mobile interface immediately upon the successful completion of the errand to enforce data minimization principles. Cultural and Political Requirements: Requirement ID,Non-Functional Specification REQN017,"The system interface and all auto-generated digital waybills shall utilize English as the primary operational language, supplemented by localized terminology (e.g., Pabili, Padala)." REQN018,The system shall consistently format all financial calculations and digital invoices utilizing the Philippine Peso (PHP) currency designation. REQN019,The system shall record and display all temporal data utilizing the standard 24-hour time format to eliminate AM/PM ambiguity during multi-shift dispatching. REQN020,The system shall structure all chronological reports and date inputs utilizing the MM/DD/YYYY standard format.
